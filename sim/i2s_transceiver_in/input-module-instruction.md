# transceiver_feeder — Specification for Implementation

## Overview

Create a Verilog module `transceiver_feeder` and a companion Python script that together provide realistic I2S stimulus to `transceiver_in` for simulation. The feeder reads pre-digested sample data from a `.hex` memory file (generated by the Python script) and drives the `ws` and `sd` lines with proper I2S timing. After exhausting the sample file it pads with zeroes forever and asserts a `done` signal.

## I2S Timing (Proper)

Reference: standard I2S Philips protocol.

| Parameter | Value |
|---|---|
| Sample rate (`fs`) | 48 kHz |
| Bit clock (`sck`) | 64 × fs = 3.072 MHz |
| Bits per slot | 32 (24 data + 8 padding) |
| Frame (L + R) | 64 SCK cycles |
| Data valid | MSB-first, 24-bit, left-justified within slot |
| Data change edge | **negedge** of SCK |
| Data capture edge | **posedge** of SCK (as `transceiver_in` already samples on posedge) |
| WS change edge | **posedge** of SCK (WS changes at start of frame, aligned with MSB) |

Timing diagram (one frame of 64 SCK cycles):

```
SCK   [____|‾|_|‾|_|‾|_|‾|_|‾|_|‾|_|‾|_|‾|_|‾ ... |‾|____]
      posedge   ^   ^   ^   ^   ^   ^   ^   ^   ... ^   ^     (transceiver_in samples here)
      negedge  ^   ^   ^   ^   ^   ^   ^   ^   ^   ... ^     (feeder changes sd here)

WS    [_______________ LEFT _______________|_______________ RIGHT ______________]
      change on posedge #0                    change on posedge #32

SD    [b23|b22|b21|...|b0|0|0|0|0|0|0|0|0|b23|b22|...]
       ‾‾‾‾‾‾‾‾ 24 data bits ‾‾‾‾‾‾‾‾‾‾‾ 8 zero pad ‾‾‾‾‾‾‾‾ 24 data bits ‾‾‾‾‾
```

Key rules:
- `sd` changes on **negedge** of `sck`, not posedge.
- `ws` changes on **posedge** of `sck` (it's synchronous to the bit clock and changes at the start of each slot).
- The first data bit of each slot is driven onto `sd` on the **negedge immediately following** the posedge where WS changed. This means the first `sd` transition happens at negedge #0.5 (half a clock after WS change), giving the receiver a full half-cycle setup before the next posedge.

## Module: `transceiver_feeder`

### Ports

```verilog
module transceiver_feeder #(
    parameter SCK_HALF_PERIOD = 163,   // [(1e9 / 48000 / 64) / 2] in ns ≈ 163 ns for 3.072 MHz
    parameter NUM_SAMPLES     = 0      // 0 = infer from file size; >0 = force count
)(
    // Control
    input  wire         clk,       // system clock (must be fast enough to time SCK edges)
    input  wire         rst_n,     // active-low async reset

    // Status
    output reg          done,      // asserted when all samples exhausted (stays high)

    // I2S output to transceiver_in
    output reg          sck,       // bit clock (3.072 MHz)
    output reg          ws,        // word select (L/R)
    output reg          sd         // serial data (changes on negedge of sck)
);
```

### Parameters — Configurable at Top

```verilog
// ══════════════════════════════════════════════════════════════════════════════
//  USER PARAMETERS — tweak these for different sample rates / data
// ══════════════════════════════════════════════════════════════════════════════
parameter SCK_HALF_PERIOD = 163,   // SCK half-period in ns @ 48 kHz fs, 64× fs
                                    // Formula: (1_000_000_000 / fs / 64) / 2
                                    // For 48k: 163 ns
                                    // For 96k: 82 ns
parameter NUM_SAMPLES     = 0,      // Number of samples in the .hex file.
                                    // Set to 0 to have the module count samples
                                    // automatically (reads until EOF marker).
                                    // Set >0 to force a specific count.
// ══════════════════════════════════════════════════════════════════════════════
```

### Internal Behaviour

1. **Clock generation:** Drive `sck` by toggling every `SCK_HALF_PERIOD` ns. Since Verilog sims use `#delay`, the `clk` input is actually the *timing reference* — the module shouldgg simply `always #SCK_HALF_PERIOD sck = ~sck;` style timing. Actually, simpler: use a counter based on the system clock (`clk`) to generate SCK edges. *Alternative:* Since this is simulation-only, use `#delay` timing inside the module directly (delay-based SCK generation).

   **Preferred approach for simulation:** Use `initial` blocks and `#delay` to generate SCK and sequence the I2S frame. This is simpler and more readable for simulation-only code. Wrap it all in an initial block that runs once and loops.

2. **Sample storage:** Use `reg signed [23:0] sample_mem [0:MAX-1];` loaded via `$readmemh("samples.hex", sample_mem);`. The `done` signal starts low and goes high when all samples have been transmitted and the module enters zero-padding mode.

3. **Frame sequencing per sample (mono → stereo):**
   For each sample in memory:
   - Drive `ws` LOW (left channel) for 32 SCK cycles.
   - During left slot: drive `sd` with the sample bits MSB-first on negedge, then 8 zero bits.
   - Toggle `ws` HIGH (right channel) for 32 SCK cycles.
   - During right slot: drive `sd` with **the same sample bits** MSB-first on negedge, then 8 zero bits.
   - Wait 1 SCK cycle? No — frames are back-to-back, no gap.

4. **After all samples exhausted:**
   - `done` <= 1
   - Continue sending zeroes on both channels forever.
   - `sd` stays low, `sck` and `ws` continue toggling.

5. **Reset behaviour:**
   - `rst_n = 0`: stop SCK, set `ws=0`, `sd=0`, `done=0`.
   - On de-assertion, re-read the memory file and start from the beginning.

### Memory File Loading

Use `$readmemh("samples.hex", sample_mem)`. The `.hex` file must be in the same directory as the simulation working directory (or use a full path). Stick with the convention that `iverilog` is run from the directory containing the testbench, so a relative path `"samples.hex"` works.

#### File format (`samples.hex`)

One 6-character hexadecimal value per line (24-bit, signed), most-significant nibble first:

```
000000
7FFFFF
800000
123456
```

These are read by Verilog's `$readmemh` which interprets them as 2's-complement signed values when connected to a `reg signed` array. The module should declare the memory as:

```verilog
reg signed [23:0] sample_mem [0:SAMPLE_MAX-1];
```

Instantiate both `transceiver_feeder` and `transceiver_in` together. Connect:
- `transceiver_feeder.sck` → `transceiver_in.clk` (bit clock serves as transceiver clock)
- `transceiver_feeder.ws` → `transceiver_in.ws`
- `transceiver_feeder.sd` → `transceiver_in.sd`
- `transceiver_in.o_tready` → `1'b1`

The testbench should:
- Dump VCD to `feeder_tb.vcd`
- Run `transceiver_feeder` through several frames
- Wait until `done` goes high
- Display received words via `$display` and compare to expected
- `$finish` some time after done

Key check: `transceiver_in.tdata` should match the original 24-bit hex values from `samples.hex` (accounting for pipeline delay through the I2S frame).

## Python Script: `gen_samples.py`

### Purpose

Reads user configuration, generates a `.hex` file consumed by `$readmemh`.

### Interface

```bash
python3 gen_samples.py                    # uses UI prompts, writes samples.hex
python3 gen_samples.py --output out.hex   # custom output path
```

### Guitar Noise Generator

The script should produce a waveform that imitates a guitar string — it doesn't need to be a full physical model, but should sound/feel like a plucked note when played back. Think of it as a "budget guitar noise maker for test bench."

#### User-Facing Configuration (at the top of the Python file, clearly labelled)

```python
# ══════════════════════════════════════════════════════════════════════════════
#  USER PARAMETERS — tweak these to generate different tones
# ══════════════════════════════════════════════════════════════════════════════

# Fundamental frequencies in Hz (standard guitar tuning by default)
# E2=82.41, A2=110.00, D3=146.83, G3=196.00, B3=246.94, E4=329.63
FREQUENCIES_HZ = [82.41, 110.00, 146.83, 196.00, 246.94, 329.63]

# Mix weights for each frequency (amplitude, 0.0–1.0)
# Default: strum all strings equally
FREQUENCY_GAINS  = [0.4, 0.4, 0.5, 0.6, 0.7, 0.8]

# Decay envelope: how fast each frequency fades (per-sample multiplier)
# 1.0 = no decay, 0.999 = slow decay, 0.99 = fast decay
DECAY_RATES      = [0.9995, 0.9995, 0.9990, 0.9985, 0.9980, 0.9970]

# Noise component: adds a percussive "pick" transient
# The noise fades quickly — simulates the attack of a plectrum
NOISE_GAIN       = 0.15        # initial noise amplitude (0.0–1.0)
NOISE_DECAY      = 0.98        # per-sample noise decay multiplier

# Output format
SAMPLE_RATE      = 48000       # Hz
NUM_SAMPLES      = 48000       # 1 second of audio at 48 kHz
BIT_DEPTH        = 24          # bits per sample

# ── Generated sound type ──
# "pluck"  = Karplus-Strong-like string pluck (default, sounds like guitar)
# "chord"  = sum-of-sines with decay (cleaner, more synthetic)
# "arpeggio" = frequencies ring in one at a time
SOUND_TYPE = "pluck"

# ══════════════════════════════════════════════════════════════════════════════
```

#### Sound generation modes

**"pluck" mode (default):** For each frequency, generate a sine wave at that frequency, then apply an exponentially decaying envelope. Sum all frequencies together with their gains. Add a white noise burst at the start (decaying rapidly) to simulate pick attack. Clip to 24-bit signed range.

Algorithm:
```
output[n] = sum_over_freqs(freq_gain[f] * sin(2π * freq[f] * n / fs) * decay[f]^n)
            + noise[n] where noise[n] = noise_gain * noise_decay^n * uniform_random(-1, 1)
```

**"chord" mode:** Same as pluck but without noise component — cleaner sum-of-sines.

**"arpeggio" mode:** Frequencies fade in sequentially. Frequency `i` starts at sample `i * (NUM_SAMPLES / len(frequencies))` with its own attack envelope.

#### Output

Write one line per sample in the `.hex` file, each being exactly 6 hex digits (24-bit two's complement, uppercase). The script should:

1. Compute the waveform
2. Clip to signed 24-bit range [-2^23, 2^23-1]
3. Convert to two's complement hex
4. Write `samples.hex`

#### Additional features

- Print a summary to stdout: sample rate, number of samples, duration, peak amplitude, RMS
- Allow `--seed N` for reproducible noise (useful for comparing simulation runs)
- Print a warning if clipping occurred and how many samples clipped

### Optional: `frequencies.json`

If the user provides a `frequencies.json` file in the same directory, the script can load frequencies/gains/decay rates from it instead of using the hardcoded defaults. Format:

```json
{
    "frequencies_hz": [82.41, 110.00],
    "frequency_gains": [0.5, 0.5],
    "decay_rates": [0.999, 0.999],
    "noise_gain": 0.15,
    "noise_decay": 0.98,
    "sound_type": "pluck"
}
```

If no JSON file exists, defaults in the script head are used.

## Implementation Notes

### Verilog

- **Pure Verilog, no SystemVerilog.** Must compile with `iverilog`.
- Active-low synchronous reset (`rst_n`).
- `snake_case` naming.
- Only use `$readmemh` for file I/O — no `$fopen`, `$fread`, etc. (simpler, fewer issues with iverilog).
- The `sck` generation should use `#delay` inside an `initial` block since this is simulation-only. No PLLs, no clock dividers.
- Drive `sd` on negedge: use `always @(negedge sck)` block or sequence it with delays.
- Comment generously, especially the top-level parameter block and the I2S timing state machine.
- The feeder must properly handle the one-cycle `sd` pipeline delay through `transceiver_in` (the receiver latches `sd` on posedge, but starts sampling the MSB immediately after a WS edge — verify with `transceiver_in_tb.v` that the alignment is correct).

### Python

- Python 3 only, standard library dependencies only (`argparse`, `math`, `json`, `random`).
- No numpy/scipy requirement — keep it zero-dep.
- If the user wants to add those later, the structure should support it (but don't require them).

### Testbench

- Dump VCD to `feeder_tb.vcd`
- Run for at least 3 complete I2S frames
- Print received words vs expected with `$display`
- Exercise the `done` signal: wait until it goes high, then run another 100 SCK cycles to verify zero-padding behaviour
